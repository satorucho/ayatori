# 判断基準（品質チェック観点）

生成後のセルフレビュー、およびレビューモードで使用するチェックリスト。
全項目を順番に検証し、1つでも違反があれば修正してから出力する。

---

## 優先度

**最優先4項目は省略不可。省略した場合、品質チェック未実施と同等の扱いとする。**

| 優先度 | 項目 | 内容 |
|--------|------|------|
| 最優先 | A-7 | 分岐合流の検証 |
| 最優先 | B-1 | 矢印ギャップの検証 |
| 最優先 | C-1 | 計算表の存在確認 |
| 最優先 | D-2 | 最終走査の検証 |
| 標準 | A-1〜A-6, A-8 | フロー構造チェック |
| 標準 | B-2〜B-5 | スペーシングチェック |
| 標準 | C-2〜C-7 | 図形サイズチェック |
| 標準 | D-1, D-3, D-4 | 境界・レイアウトチェック |
| 標準 | E-1〜E-4 | 表現品質チェック |
| 拡張 | Phase帯関連 | Phase帯使用時のみ |
| 拡張 | ページ分割関連 | page-split.md使用時のみ |

## A. フロー構造チェック（8項目）

### A-1: Yes↓ / No→ 厳守
- [ ] すべてのダイヤモンドで Yes が下方向に進んでいるか
- [ ] すべてのダイヤモンドで No が右方向に進んでいるか

### A-2: ダイヤモンドにはYes/Noで答えられる状態・条件
- [ ] ダイヤモンド内テキストがYes/Noで回答できる形か
- [ ] 「〜済か」「〜か」「〜ないか」など状態を問う形か

### A-3: メインフローが下方向直進
- [ ] 多数派・正常系のフローがYes↓に直進しているか
- [ ] 問いの向きを逆にすることで右展開を減らせないか
- [ ] 不要な中間ノードがないか（ダイヤモンドに直結できるなら省く）
- [ ] 「やるかスキップか」分岐の場合、text-flow-template.mdの判定フローに従っているか
- [ ] No→側にL字合流線（右→下→左）が発生していないか。発生している場合、問いの反転で解消できないか確認する

### A-4: Yes/Noラベル位置
- [ ] Yesラベルがダイヤモンド下辺の直下に配置されているか
- [ ] Noラベルがダイヤモンド右辺の直右に配置されているか

### A-5: 「？」禁止
- [ ] ダイヤモンド内・ラベルに疑問符「？」が含まれていないか

### A-6: 列構成の検証
- [ ] 分岐No→の先に2ステップ以上ある場合、列として展開されているか
- [ ] 1つの列に対して、接続元の分岐が1つだけか（1列1接続原則）
- [ ] ネストが3列以内に収まっているか

### A-7: 分岐合流の検証（★最優先）
- [ ] No→の合流Y座標が、Yes側の最終ノード下端+64pxの位置にあるか
- [ ] 計算表の合流Y列の値とSVGのY座標が一致するか
- [ ] No→の縦線のX座標が、Yes側の全ノードの最大右端+30px以上離れているか
- [ ] No→の合流線がYes側のrect/polygon/ellipseの内部を通過していないか

### A-8: 参照ノードの検証
- [ ] 参照ノード（同色）がある場合、参照元と参照先で同じテキスト・同じ色か
- [ ] 参照先のフローが実際に存在するか

## B. スペーシングチェック（5項目）

### B-1: 矢印ギャップの検証（★最優先）
- [ ] 全垂直矢印のy1が前ノード下端+6になっているか
- [ ] 全垂直矢印のy2が次ノード上端-14になっているか
- [ ] 全水平矢印のx1がノード右端+6になっているか
- [ ] 全水平矢印のx2が次ノード左端-14になっているか
- [ ] 座標表の上端/下端/左端/右端にギャップが混入していないか
- [ ] marker補正-8pxが終点にのみ適用されているか

### B-2: 矢印長パターンの統一
- [ ] 通常の垂直矢印が全てパターンM（64px間隔）か
- [ ] 水平矢印（No→）が全てパターンM水平（80px間隔）か
- [ ] パターンLが時間軸整合・列展開にのみ使用されているか
- [ ] パターンLのノード端間隔が84px以上あるか

### B-3: 同種矢印の長さ一致
- [ ] 同じフロー内の全M垂直矢印が64px間隔か
- [ ] 同じフロー内の全M水平矢印が80px間隔か

### B-4: 図形同士の重なり・交差なし
- [ ] ノードが重なっていないか
- [ ] 矢印が意図しないノードを横切っていないか
- [ ] 列0の短い枝が列1のノードと横方向で重なっていないか

### B-5: 列間の干渉検証
- [ ] 列0の短い枝の右端と列1の左端の間に40px以上の間隔があるか
- [ ] 列間の水平矢印が他のノードを横切っていないか

---

## C. 図形サイズチェック（7項目）

### C-1: 計算表の存在確認（★最優先）
- [ ] テキスト計算表が作成されているか
- [ ] 図形サイズ表が作成されているか
- [ ] 座標表が作成されているか
- [ ] 上記3表なしにSVGが生成されていないか

### C-2: 計算表の網羅性
- [ ] 計算表の行数がSVGのノード数（短い枝を含む）と一致するか
- [ ] 全ノードに全角/半角のカウントが記載されているか

### C-3: テキスト凍結の検証
- [ ] SVGの全textタグのテキストが計算表と完全一致するか
- [ ] 計算表作成後にテキストを変更した場合、再計算が行われたか

### C-4: 図形サイズの正確性
- [ ] 全角/半角が正しくカウントされているか
- [ ] ダイヤモンドの×1.42が適用されているか
- [ ] 最小サイズが適用されているか
- [ ] 同一列内の矩形幅が統一されているか

### C-5: テキストの収まり
- [ ] 1行テキストの幅が矩形幅-48pxを超えていないか
- [ ] 全ノードのテキストが図形内に収まっているか
- [ ] ダイヤモンドテキストが2行×8全角文字以内か
- [ ] 楕円テキストが2行以内か

### C-6: 座標表とSVGの整合性
- [ ] SVGの全rect座標が座標表と一致するか
- [ ] SVGの全polygon座標が座標表と一致するか
- [ ] SVGの全ellipse座標が座標表と一致するか

### C-7: CY計算の正確性
- [ ] 全ノードのCYが「前ノードの下端+64+図形上端からCYまでの距離」で計算されているか
- [ ] CYからの逆算が使われていないか

---

## D. 境界・レイアウトチェック（4項目）

### D-1: 境界ボックスの検証
- [ ] 境界ボックス計算表が作成されているか
- [ ] 最大右端が座標表の右端列のmax値と一致するか

### D-2: 最終走査の検証（★最優先）
- [ ] SVG生成後に最終走査が実施されたか
- [ ] 全rect要素のx+widthがSVG幅-40以下か
- [ ] 全polygon要素の全頂点x座標がSVG幅-40以下か
- [ ] 全ellipse要素のcx+rxがSVG幅-40以下か
- [ ] 全要素の下端がSVG高さ-40以下か
- [ ] viewBox/width/heightが最終走査の値と一致するか

### D-3: レーン干渉の検証
- [ ] 各レーン内の全要素の右端がレーン区切り線Xを超えていないか
- [ ] 区切り線X = (前レーン最大右端 + 次レーン最小左端) / 2 になっているか
- [ ] レーン区切り線を挟んで左右の余白が均等か

### D-4: 修正後の再計算検証
- [ ] 座標修正後に境界ボックス計算表が再作成されたか
- [ ] viewBox/width/heightが再計算後の値と一致するか

---

## E. 表現品質チェック（4項目）

### E-1: カラーパレット準拠
- [ ] 全ノードの fill/stroke/text が style-guide.md のカラーパレットに準拠しているか
- [ ] SVG内に`#666`が使用されていないか（`#999`を使用すること）

### E-2: フォント体系準拠
- [ ] 全テキストが24/20/16/12pxの4パターンのみか
- [ ] フォントウェイトが規定通りか
- [ ] Noto Sans JPが指定されているか

### E-3: SVGコメント
- [ ] 各ノードに座標コメントがあるか

### E-4: 凡例
- [ ] 使用しているノード種別が全て凡例に含まれているか
- [ ] 参照ノードがある場合、凡例に含まれているか
