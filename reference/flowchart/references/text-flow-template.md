# テキストフロー標準フォーマット

flow-designerの出力形式。この形式以外で出力することを禁止する。
flowchartスキルはこの形式を入力として受け取る。

---

## フォーマット

```markdown
# テキストフロー: {フロー名}

## メタ情報
- フロー名: {名称}
- 目的: {「〜が完了するまで」で終わる1文}
- 粒度: {経営層向け / 業務担当向け / エンジニア向け}
- レーン: {アクター1} / {アクター2} / ...

## フロー構造

開始（{開始条件}）
  ↓
{処理1}（{実施場所/システム}）
  ↓
分岐①: {Yes/Noで答えられる問い}
  No → {短枝の処理}
  Yes↓
{処理2}
  ↓
完了

## 設計判断メモ
- 分岐①を「{問い}」にした理由: {理由}
- {参照ノードの採用理由}

## 要確認事項
①{確認事項}
②{確認事項}
```

---

## 記法ルール

| 記法 | 意味 | 例 |
|------|------|-----|
| `↓` | 次の処理への遷移（垂直） | `処理A ↓ 処理B` |
| `分岐①:` | 分岐ダイヤモンド（通し番号） | `分岐①: 個人決済か` |
| `No →` | 分岐のNo方向（右方向=例外系） | `No → 処理B` |
| `Yes↓` | 分岐のYes方向（下方向=正常系） | `Yes↓` |
| `【{名前}】` | サブフローのラベル | `【都度払いフロー】` |
| `（{補足}）` | 補足テキスト（実施場所等） | `（受注システム上で実施）` |
| `※` | 業務ルールの補足 | `※ 週次/月次が企業ごとに設定` |
| `完了` / `終了` | フローの終端 | |

### 分岐番号のルール

- 通し番号（①②③...）を使用する
- 番号を飛ばさない（①→③はNG）
- 【分岐ラベル】に「分岐①のNo→」のように接続元を明記する

### 分岐の書き方ルール（No→先書き）

**分岐では必ずNo→（短枝）を先に書き、Yes↓（メインフロー）を下に続ける。**

```
分岐①: 現金回収が必要か
  No → 回収不要→終了
  Yes↓
1軒1軒現金回収する
  ↓
次の処理...
```

理由：Yes↓が下に続くメインフローなので、先にNo→の短枝を書き切ってからメインフローを書いた方が読みやすい。Yes↓のメインフローを全部書いた後にNo→が出てくると、分岐との距離が離れすぎて対応関係が読みにくくなる。

### インデントのルール

- 分岐のNo→/Yes↓は2スペースインデント
- サブフロー内の処理はさらに2スペース追加（計4スペース以上）
- ↓は処理と同じインデントレベル

## 「やるかスキップか」分岐の設計ガイド

準備作業・更新作業など「必要なら実行、不要ならスキップ」という分岐の設計ルール。

### 判定フロー

分岐を設計する際、以下の順で判定する。

```
1. 実行側（Yes）が多数派か？
   → Yes: 通常ルール適用。「〜済みか」「〜問題ないか」でYes↓=実行。
   → No: 2へ。

2. 実行側（No→右）に後続処理が2つ以上あるか？
   ※ ステップ数はノードの数で数える。合流線は数えない。
   → Yes: 通常ルール適用。「〜が必要か」でYes↓=実行。
         No→右はスキップ1本で済むため、L字合流線が
         右側（短い方）に発生する形の方がシンプル。
   → No: 3へ。

3. 「〜は不要か」または同等の自然な問いに書けるか？
   → Yes: 反転パターン適用。Yes↓=スキップ、No→=実行。
   → No: 通常ルール適用。問いの文言を工夫する。
         例:「〜に変更はないか」「〜は最新の状態か」
```

### 反転パターンの適用条件（3条件すべて満たす場合に適用）

1. スキップ（何もしない）が多数派である
2. 実行側の処理が1つで完結する（後続処理の連鎖がない）
3. 問いの文言が業務的に自然である

### 適用例

**OK: 反転パターン適用**
- 「カレンダー更新は不要か」→ Yes↓=スキップ / No→=更新
- 「従業員情報は最新の状態か」→ Yes↓=スキップ / No→=更新作業

**OK: 通常ルール適用**
- 「登録内容に誤りはないか」→ Yes↓=正常系（誤りなし）/ No→=修正
- 「承認済みか」→ Yes↓=次へ / No→=承認処理

**NG: 反転パターンを誤適用**
- ×「承認は不要か」→ 承認はやるべき作業。「不要か」が不自然。
- ×「カレンダー更新は不要か→No→更新→通知→確認」→ 実行側に後続処理が複数。

## 出力例（会計処理フロー）

```markdown
# テキストフロー: 会計処理フロー

## メタ情報
- フロー名: 会計処理フロー
- 目的: 月次の入金確認・請求処理が完了するまで
- 粒度: 業務担当向け
- レーン: 玉子屋管理者 / 企業管理者

## フロー構造

開始（月末〜翌月）
  ↓
当月の売上データを集計する（受注システム）
  ↓
分岐①: 新システム経由の注文か
  No → 電話FAX経由：分岐③へ
  Yes↓

【新システム経由：分岐①のYes→】
分岐②: 個人決済か
  No → 【企業払い】まとめ払い処理 ※ まとめ払いフローと同一（参照ノード）
  Yes↓

【個人決済フロー】
決済代行会社の管理画面/メールで入金明細を確認する
  ↓
決済手段ごとに売上データと突合する
  ↓
分岐⑥: 金額・件数・手数料に差異はないか
  No → 決済代行に修正依頼
  Yes↓
入金額を確認する（手数料差引後）
  ↓
手数料の会計処理を行う
  ↓
完了

【電話FAX経由：分岐①のNo→】
分岐③: 都度払いか
  No → 【まとめ払いフロー】
         請求サイクルに応じて請求書を発行する（オービック経由）
         ※ 週次払い・月次払いのいずれかが企業ごとに設定されている
           ↓
         企業からの入金を確認する
           ↓
         分岐⑤: 未入金の企業はないか
           No → 督促連絡
           Yes↓
         完了
  Yes↓

【都度払いフロー】
回収金額の月次集計を確認する
  ↓
売上データとオービックを突合する
  ↓
分岐④: データに差異はないか
  No → 差異を調査・修正
  Yes↓
完了

## 設計判断メモ
- 分岐①を「新システム経由か」にした理由: 都度払いは電話FAX経由のみのため、最上位で経路を分ける
- 企業払いは参照ノードにした理由: まとめ払いフローと処理が同一であり、列を増やすと横幅が過大になるため
- 全分岐でNo→を先に書き、Yes↓メインフローを下に続ける形式を採用

## 要確認事項
①都度払いの回収金額の集計方法（受注システムか紙ベースか）
②個人決済の売上データ突合の実施場所（受注システムか、新システム管理画面に機能を設けるか）
③オービックで決済手段別の入力・管理が可能か
```
